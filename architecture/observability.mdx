RAGler is designed to be observable in production. All critical lifecycle transitions emit structured JSON log events. Prometheus-style metrics are defined for tracking throughput and latency across all layers.

## Structured logging

Logs are emitted as JSON objects via the NestJS `Logger`. Every log entry includes an `event` key that identifies the transition, plus contextual fields such as `sessionId`, `userId`, `collectionId`, `durationMs`, and error details.

<Note>
  Never log secrets. The `OPENAI_API_KEY` and any user-supplied API key overrides must never appear in log output. The Settings module masks keys before storing them in response DTOs.
</Note>

## Prometheus metrics

The following Prometheus-style counters, histograms, and gauges are the recommended metric set for a RAGler deployment. Instrument these via your preferred NestJS metrics middleware (e.g. `@willsoto/nestjs-prometheus`).

### HTTP layer

| Metric | Type | Labels |
|--------|------|--------|
| `http_server_requests_total` | Counter | `route`, `method`, `status` |
| `http_server_request_duration_seconds` | Histogram | `route`, `method` |

### Dependency calls

| Metric | Type | Labels |
|--------|------|--------|
| `dependency_requests_total` | Counter | `dep` (redis/qdrant/openai), `op`, `status` |
| `dependency_request_duration_seconds` | Histogram | `dep`, `op` |

### LLM

| Metric | Type | Labels |
|--------|------|--------|
| `llm_requests_total` | Counter | `scenario` (chunking/embedding/tagging/refine), `status` |
| `llm_request_duration_seconds` | Histogram | `scenario` |

### Publish operations

| Metric | Type | Labels |
|--------|------|--------|
| `publish_operations_total` | Counter | `status` (success/failure) |

### Draft sessions

| Metric | Type | Labels |
|--------|------|--------|
| `draft_sessions_active` | Gauge | (none) |

## Health check endpoints

Two health endpoints are available for orchestration probes. See the [Health module](/architecture/modules/health) for full response shapes.

| Endpoint | Purpose | Use as |
|----------|---------|--------|
| `GET /api/health/liveness` | Process alive | Kubernetes `livenessProbe` |
| `GET /api/health/readiness` | Redis + Qdrant reachable | Kubernetes `readinessProbe` |
| `GET /api/health` | Full health with details | Manual monitoring |

## Recommended dashboard

When building a monitoring dashboard, the following counters and queries provide a clear operational view:

- **Ingest rate** — count of `ingest_start` events per minute, by `sourceType`
- **Publish success/failure rate** — `publish_success` and `publish_failure` counts over time
- **Publish latency p50/p95** — `durationMs` from `publish_success` events
- **Search latency p50/p95** — `durationMs` from `search_complete` events
- **Active draft sessions** — `draft_sessions_active` gauge
- **LLM error rate** — `chunking_failure` + `embedding_batch_failure` counts
- **Collection editor operations** — `chunk_created` + `chunk_updated` + `chunk_deleted` per collection

## Design principles

<AccordionGroup>
  <Accordion title="Stateless HTTP service">
    The NestJS backend holds no in-process mutable state between requests. All session state is externalised to Redis (drafts) or Qdrant (published knowledge). This means the service can be restarted, redeployed, or scaled horizontally without data loss.
  </Accordion>

  <Accordion title="Dual-store model">
    Two distinct stores serve different purposes:

    - **Redis** — ephemeral, TTL-bound draft sessions. Fast reads and writes during the editing phase. Sessions are deleted on publish or explicit user action.
    - **Qdrant** — durable, indexed vector store for published knowledge. The `sys_registry` pseudo-collection holds collection metadata as Qdrant points with a scalar `[0]` vector.
  </Accordion>

  <Accordion title="Atomic publish replacement">
    Publishing a session replaces all previous vectors for that `doc.source_id` in a two-step sequence: delete by filter, then upsert. Embeddings are generated **before** the delete step. If embedding generation fails, the existing Qdrant data is untouched. If the upsert fails after a delete, the session is not removed from Redis, allowing a retry.
  </Accordion>

  <Accordion title="Human-in-the-loop (HITL) gate">
    LLM output is always proposal-based. Chunking produces a draft that a human reviews before any data reaches Qdrant. The session lifecycle enforces this: chunks can only be edited in `DRAFT` status, the `PREVIEW` step locks editing, and only an explicit publish action writes to the vector store.
  </Accordion>

  <Accordion title="DTO validation at the boundary">
    All incoming request payloads are validated by Zod schemas via `nestjs-zod` and the global `ZodValidationPipe`. No raw input reaches business logic without schema validation. The same Zod types are used for TypeScript inference throughout the codebase.
  </Accordion>
</AccordionGroup>

## Module boundaries

Each NestJS module owns a single well-defined responsibility. Cross-module access goes through exported services — never direct infrastructure calls from a module that does not own that store.

| Module | Owns | Depends on |
|--------|------|------------|
| `ingest` | Session creation, source normalisation | `LlmModule`, `RedisService`, ingest strategies |
| `session` | Draft chunk lifecycle, preview, publish | `IngestService`, `LlmService`, `QdrantClientService` |
| `collection` | Collection CRUD, chunk editor operations | `QdrantClientService`, `LlmService` |
| `vector` | Vector search | `QdrantClientService`, `LlmService` |
| `llm` | OpenAI calls (chunking, embeddings, refinement) | `SettingsService`, OpenAI SDK |
| `settings` | Agent configuration persistence | `RedisService` |
| `health` | Liveness and readiness probes | `RedisService`, `QdrantClientService` |
| `mcp` | IDE integration via Model Context Protocol | `VectorService`, `CollectionService` |

## API structure

```
Global prefix:  /api
Swagger UI:     /api/docs
MCP server:     :3100/mcp  (separate HTTP server, no NestJS middleware)
```

The REST API uses the `X-User-ID` header to propagate user identity into service calls. There is no authentication middleware in the baseline; the header is read and passed to services where needed for audit purposes.

## Request handling

```
HTTP request
    │
    ▼
Helmet (security headers)
    │
    ▼
ZodValidationPipe (parse + validate all DTOs)
    │
    ▼
Controller (thin routing layer)
    │
    ▼
Service (business logic)
    │
    ├── Redis (session state)
    └── Qdrant (vector store)
             │
             └── OpenAI API (LLM calls)
```

Errors are mapped by `HttpExceptionFilter` to a consistent JSON shape before they reach the client.

## Storage model detail

<Tabs>
  <Tab title="Redis">
    Sessions are stored as JSON strings under the key `session:{sessionId}`. The TTL is configured via `session.ttl` (environment variable). Sessions are scanned with the pattern `session:session_*`.

    ```
    Key:    session:session_<uuid>
    Value:  JSON (SessionData)
    TTL:    configurable (e.g. 86400 seconds)
    ```

    Settings are stored under a fixed key with no TTL:

    ```
    Key:    settings:agent
    Value:  JSON { modelName, apiKey }
    TTL:    none
    ```
  </Tab>
  <Tab title="Qdrant">
    Published chunks live in per-collection Qdrant collections named `kb_{collectionId}`. The collection registry is a special collection named `sys_registry`.

    ```
    sys_registry          — collection metadata (name, description, created_by)
    kb_{collectionId}     — published chunk vectors + payload
    ```

    Each `kb_*` collection has payload indexes on:
    - `doc.source_type` (keyword)
    - `doc.source_id` (keyword)
    - `chunk.type` (keyword)
    - `chunk.lang` (keyword)
    - `tags` (keyword)

    Embeddings use `text-embedding-3-small` (1536 dimensions).
  </Tab>
</Tabs>

## Concurrency and safety

| Concern | Behaviour |
|---------|----------|
| Concurrent edits to same session | Last write wins. Redis `setJson` is not transactional. Concurrent edits to the same session from multiple tabs may produce partial state. |
| Publish while editing | `PREVIEW` status blocks further edits. The client must call `/preview` before `/publish`. |
| Duplicate publish | Atomic replacement is idempotent: publishing the same source twice replaces the same `doc.source_id` filter. |
| Embedding failure mid-publish | Qdrant delete has not yet run, so existing data is preserved. The session remains in Redis for retry. |
| Pod restart | Redis and Qdrant are external. No state is lost on process restart. |

## Graceful shutdown

The application registers `SIGTERM` and `SIGINT` handlers. On signal receipt:

1. The MCP HTTP server stops accepting new connections.
2. All active MCP transport sessions are closed.
3. The NestJS application closes (drains in-flight requests via `enableShutdownHooks`).
4. A 30-second timeout forces exit if shutdown is not completed.
